# HappyRobot Configuration Notes

## Boundary Decision

Reuse the existing HappyRobot workflow pieces that already work:

- Web call trigger
- Inbound Voice Agent
- Existing FMCSA carrier lookup AWS webhook
- Existing load lookup AWS webhook

Build externally only where the challenge explicitly asks for differentiated implementation:

- Negotiation decision API
- Call event ingestion
- Post-call summary ingestion
- Custom metrics dashboard
- Dockerized Railway deployment story

This avoids duplicating the existing AWS FMCSA/load services and keeps the demo focused on the missing product surface.

Use this prompt inside the HappyRobot docs or workflow assistant:

```text

Please help me update the workflow for the FDE inbound carrier sales challenge.

I do not want to rebuild the existing FMCSA or load lookup webhooks. I want to add Railway-hosted FastAPI endpoints:
- POST /api/offers/evaluate
- POST /api/calls/events
- POST /api/calls/summary
- GET /api/metrics
- GET /dashboard

Please show me exactly where in the HappyRobot UI to:
1. Upgrade the voice agent model from deprecated gpt-4o Turbo to a current recommended model.
2. Complete the prompt logic after verify_carrier so the agent evaluates whether the FMCSA result means the carrier is eligible.
3. Make the agent call find_available_loads after it has a reference number, or after it has enough lane/equipment details.
4. Add a new external API tool called evaluate_offer that calls POST /api/offers/evaluate with an x-api-key header.
5. Add negotiation instructions so the agent handles up to 3 carrier counter-offers.
6. Add a mock transfer message when a price is agreed: "Transfer was successful and now you can wrap up the conversation."
7. Enable sentiment classification for positive, neutral, and negative.
8. Fix the call outcome classifier typo from "Sucess" to "Success" and expand outcomes.
9. Expand the AI Extract node to capture the full call summary schema.
10. Add a final Webhook node that sends extraction and classifier results to POST /api/calls/summary with an x-api-key header. The x-api-key value should be generated by the Railway API key admin endpoint, not manually invented.
```

## Voice Agent Prompt Patch

Add this behavior to the existing voice agent prompt:

```text
After collecting the MC number, call verify_carrier.

If the carrier verification response indicates the carrier is not eligible, inactive, out of service, missing operating authority, or missing required insurance, politely explain that we cannot book the load until compliance is resolved. Do not pitch or negotiate a load. End the call after offering to have the team follow up.

If the carrier is eligible, confirm the carrier name with the caller, then continue to load matching.

When the caller provides a reference number, call find_available_loads with that reference number. If the caller does not have a reference number, ask for origin, destination, pickup timing, and equipment type, then use the available load lookup path configured in the workflow.

When a load is found, pitch the load naturally using origin, destination, pickup time, delivery time, equipment type, commodity, weight, miles, notes, and listed rate. Ask if the carrier wants to book it at the listed rate.

If the carrier accepts the listed rate, say: "Great, we can move forward at that rate. Transfer was successful and now you can wrap up the conversation."

If the carrier makes a counter-offer, call evaluate_offer with call_id if available, load_id or reference_number, loadboard_rate, offer_rate, and negotiation_round.

Follow the evaluate_offer response exactly:
- If decision is "accept", confirm the accepted_rate and say the transfer message.
- If decision is "counter", say the returned say message and ask whether that rate works.
- If decision is "reject", politely explain that the rate will not work and close the call.

Do not negotiate more than 3 rounds. Keep the conversation concise and professional.
```

## evaluate_offer Tool

Method: `POST`

URL:

```text
https://YOUR-RAILWAY-DOMAIN/api/offers/evaluate
```

Headers:

```text
x-api-key: GENERATED_HAPPYROBOT_API_KEY
content-type: application/json
```

Request body:

```json
{
  "call_id": "{{call.id}}",
  "load_id": "{{load.load_id}}",
  "reference_number": "{{load.reference_number}}",
  "loadboard_rate": "{{load.loadboard_rate}}",
  "offer_rate": "{{carrier_offer_rate}}",
  "negotiation_round": "{{negotiation_round}}"
}
```

Expected response:

```json
{
  "decision": "counter",
  "counter_rate": 2550,
  "accepted_rate": null,
  "walkaway_rate": 2646,
  "next_action": "ask_carrier_to_accept_counter",
  "say": "I cannot get to $2,700. The best I can do right now is $2,550. Does that work?"
}
```

## Call Outcome Classifier

Replace the current three tags with:

- `Success`: carrier agreed to book and transfer was mocked successfully.
- `Rate too high`: carrier declined because rate did not work after negotiation.
- `Not eligible`: carrier could not be booked because FMCSA/compliance verification failed.
- `No matching load`: no viable load was available for the requested reference number or lane.
- `Not interested`: carrier declined for a non-rate reason.
- `No agreement`: call ended before a final booking decision.

## Sentiment Classifier

Enable either real-time sentiment on the voice agent or a separate post-call classifier with:

- `Positive`: cooperative, satisfied, or accepts booking.
- `Neutral`: factual, short, or undecided without friction.
- `Negative`: frustrated, annoyed, argumentative, or unhappy with rate/compliance outcome.
